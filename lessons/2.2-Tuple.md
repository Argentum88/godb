## Часть 1. Итоговая архитектура

### 1. `Value` — интерфейс любого значения

Центральный интерфейс. Любой объект, представляющий значение ячейки таблицы, обязан его реализовывать. Объединяет данные и поведение.

```go
// TypeID — лёгкое перечисление типов.
type TypeID int

const (
    INVALID TypeID = iota
    INTEGER
    TEXT
)

// Value — интерфейс для любого типизированного значения.
// NULL'ы за пределами Value: признак отсутствия значения живёт в Tuple.isNull,
// а сюда NULL никогда не попадает (Serialize/Compare для NULL не вызываются).
type Value interface {
    // GetTypeID возвращает ID своего типа.
    GetTypeID() TypeID

    // String возвращает человекочитаемое представление значения (для REPL/логов).
    // Идиоматичный fmt.Stringer — fmt.Println работает автоматически.
    String() string

    // Serialize дописывает байтовое представление значения в конец buf
    // и возвращает расширенный срез. Append-style избавляет от аллокации
    // на каждое значение при сборке кортежа.
    Serialize(buf []byte) []byte

    // Compare сравнивает текущий объект с другим.
    // Возвращает -1 (<), 0 (==), 1 (>). NULL'ы должны быть отсеяны выше.
    Compare(other Value) (int, error)
}
```

### 2. Конкретные реализации: `IntegerValue`, `TextValue`, …

Для каждого типа данных — отдельная структура, реализующая `Value`.

```go
type IntegerValue struct {
    value int64
}

func (iv *IntegerValue) GetTypeID() TypeID { return INTEGER }
func (iv *IntegerValue) String() string    { return strconv.FormatInt(iv.value, 10) }

func (iv *IntegerValue) Serialize(buf []byte) []byte {
    return binary.LittleEndian.AppendUint64(buf, uint64(iv.value))
}

func (iv *IntegerValue) Compare(other Value) (int, error) {
    otherInt, ok := other.(*IntegerValue)
    if !ok {
        return 0, fmt.Errorf("cannot compare INTEGER with %T", other)
    }
    switch {
    case iv.value < otherInt.value:
        return -1, nil
    case iv.value > otherInt.value:
        return 1, nil
    default:
        return 0, nil
    }
}

// По аналогии: TextValue, TimestampValue и т.д.
```

### 3. Фабрика типов: `TypeManager`

Решает проблему десериализации: по `TypeID` из схемы и `[]byte` с диска создаёт правильный `Value`.

```go
// TypeFactory — функция, создающая Value из []byte.
type TypeFactory func(data []byte) (Value, error)

// TypeManager — реестр фабрик для всех типов.
type TypeManager struct {
    factories map[TypeID]TypeFactory
}

func NewTypeManager() *TypeManager {
    factories := make(map[TypeID]TypeFactory)

    factories[INTEGER] = func(data []byte) (Value, error) {
        if len(data) == 0 {
            return nil, fmt.Errorf("empty data for INTEGER")
        }
        val := binary.LittleEndian.Uint64(data)
        return &IntegerValue{value: int64(val)}, nil
    }
    // ... фабрики для TEXT и других типов

    return &TypeManager{factories: factories}
}

func (tm *TypeManager) CreateValue(typeID TypeID, data []byte) (Value, error) {
    factory, ok := tm.factories[typeID]
    if !ok {
        return nil, fmt.Errorf("no factory for type %d", typeID)
    }
    return factory(data)
}
```

### 4. `Schema` — чертёж таблицы

Больше не хранит тяжёлые объекты `Type`, достаточно `TypeID`.

```go
type Column struct {
    Name   string
    TypeID TypeID // ключ для фабрики
}

type Schema struct {
    columns []Column
}
```

**Где хранится схема.** Объекты `Schema` живут в памяти. Их персистентное представление — в **Системном Каталоге** (отдельные служебные таблицы внутри самой БД, тема Модуля 2.4).

### 5. `Tuple` — логическое представление строки

Кортеж — это срез интерфейсов `Value`, параллельный массив `isNull` и ссылка на схему. NULL вынесен за пределы `Value` (см. интерфейс): если `isNull[i] == true`, то `values[i]` не определён и читать его нельзя.

```go
type Tuple struct {
    schema *Schema  // ссылка на схему для контекста
    values []Value  // values[i] не определён, если isNull[i] == true
    isNull []bool   // признак NULL для каждой колонки; симметричен null bitmap на диске
}

func (t *Tuple) IsNull(i int) bool {
    return t.isNull[i]
}

// Доступ к значению. Возвращает (nil, true) для NULL.
func (t *Tuple) Get(i int) (v Value, isNull bool) {
    return t.values[i], t.isNull[i]
}
```

## Часть 2. Физическая сериализация `Tuple` → `[]byte`

Формат с заголовком, по образцу PostgreSQL.

### Структура байтов на диске

```
[ ЗАГОЛОВОК КОРТЕЖА (16 байт) | NULL BITMAP (M байт) | ДАННЫЕ (всё остальное) ]
```

**Заголовок кортежа (Tuple Header)** — фиксированный размер, 24 байта.
- Назначение: метаданные MVCC (`xmin`, `xmax`, `cmin/cmax`, `infomask`, ...).
- На текущем этапе заполняется нулями — данных для них ещё нет. Расширим, когда дойдём до Модуля 8 (MVCC).

**Null Bitmap**
- Размер: 1 бит на колонку схемы (до 8 колонок — 1 байт, 9–16 — 2 байта, и т.д.).
- Если `i`-й бит = 1 → `i`-я колонка `NULL`. Это позволяет вообще не хранить `NULL`-значения в области данных.
- Симметричен `Tuple.isNull` в памяти — сериализация это просто упаковка bool'ов в биты.

**Область данных (PG-style, inline length prefix)**
- Значения уложены строго **в порядке колонок схемы**. Логический индекс колонки совпадает с порядком в области данных среди не-NULL колонок.
- Для типов **фиксированной длины** (`INTEGER`) — сериализованное значение пишется как есть (длина известна из схемы).
- Для типов **переменной длины** (`TEXT`) — сначала `uint32` (LittleEndian) с длиной значения в байтах, затем сами данные. На старте берём фиксированные 4 байта префикса; микро-оптимизация PG с 1-байтным префиксом для коротких строк — позже, при необходимости.
- NULL-значения не занимают места в области данных вообще.

**Цена выбора.** Доступ к k-й колонке — O(k) в худшем случае: чтобы добраться до неё, надо последовательно «пройти» предыдущие varlena. На наших масштабах это не проблема; при необходимости лечится кэшем смещений в `Tuple` после первого скана (как делает PG в `TupleTableSlot`).

### Алгоритм сериализации (`Tuple` → `[]byte`)

1. Создаётся пустой буфер.
2. **Заголовок:** первые 16 байт — нули (место под MVCC).
3. **Null Bitmap:** проход по `tuple.isNull`. Для каждого `isNull[i] == true` устанавливается соответствующий бит. Bitmap дописывается в буфер.
4. **Данные:** один проход по `tuple.values` и `tuple.schema` синхронно в порядке колонок.
    - Если `tuple.isNull[i] == true` — пропускаем колонку, ничего в буфер не пишем (бит уже стоит в bitmap).
    - Если тип фиксированной длины — `values[i].Serialize(buf)` дописывает значение в конец буфера.
    - Если тип переменной длины — сначала дописывается `uint32` с длиной, затем `values[i].Serialize(buf)`.

Результат — компактный `[]byte`, готовый для `SlottedPage.InsertTuple()`. Десериализация выполняется в обратном порядке: читаем заголовок (пропускаем), читаем null bitmap, дальше идём по схеме и для каждой не-NULL колонки берём либо фиксированное число байт, либо `uint32`-длину + данные, и передаём срез в фабрику `TypeManager`.

## Часть 3. Зоны ответственности и взаимодействие слоёв

> **Контекст для следующего урока.** `HeapTable` и `TableAccessMethod` будут реализованы в Занятии 2.3. Здесь описаны только для того, чтобы было ясно, *кто* и *как* будет использовать `Tuple`/`Value` — это влияет на их контракт.

### Клиент: `Executor` — но через посредника

Конечный потребитель — `Executor`. Он работает с логическими данными:
- При `INSERT` — создаёт `Tuple` из `Value`.
- При `SELECT ... WHERE` — получает `Tuple` и сравнивает его `Value`.

Но `Executor` **не должен** знать, как `Tuple` лежит на диске. `SlottedPage` или что-то другое — ему неважно. Он работает через абстрактный контракт **`TableAccessMethod`**.

### Посредник: `HeapTable` — реализация `TableAccessMethod`

`HeapTable` — мост между логическим миром `Executor` и физическим миром `SlottedPage`. Реализует интерфейс с методами:

```go
InsertTuple(tuple *Tuple) (TID, error)
GetTuple(tid TID) (*Tuple, error)
NewIterator() *TableIterator
```

Ключевой момент: `HeapTable` **принимает и возвращает логические `Tuple`, а не сырые `[]byte`**.

### Упаковщик: `HeapTable`

Именно `HeapTable` выполняет всю «грязную» работу превращения логики в физику и обратно.

**Поток `INSERT` (сверху вниз):**

1. **Executor:** создаёт логический `*Tuple` со срезом `[]Value` и передаёт в `HeapTable`.
   ```go
   tupleToInsert := NewTuple(schema, []Value{
       &IntegerValue{42},
       &TextValue{"Bob"},
   })
   heapTable.InsertTuple(tupleToInsert)
   ```
2. **HeapTable:**
    - Получает `*Tuple`. Здесь происходит **упаковка**: выполняется алгоритм сериализации (заголовок, null bitmap, укладка данных) — получается `finalTupleBytes []byte`.
    - Обращается к `BufferPoolManager` за страницей с достаточным свободным местом.
    - Вызывает `slottedPage.InsertTuple(finalTupleBytes)`.
3. **SlottedPage:** получает `[]byte` и сохраняет, ничего не зная об их смысле.

**Поток `SELECT` (снизу вверх):**

1. **Executor:** запрашивает у `HeapTable` итератор для сканирования.
2. **TableIterator** (созданный `HeapTable`):
    - Через `BufferPoolManager` получает страницу, оборачивает в `*SlottedPage`.
    - Вызывает `slottedPage.GetTuple(slotID)` → получает `[]byte`.
    - Здесь происходит **распаковка**: с помощью `TypeManager` (фабрики) и схемы байты десериализуются обратно в `*Tuple` со срезом `[]Value`.
3. **Executor:** получает готовый логический `*Tuple` и работает с его значениями.

### Схема взаимодействия

```
+----------------+
|    Executor    |  <-- Работает с логическими Tuple, Value
+----------------+
        ^
        | (вызывает методы интерфейса TableAccessMethod)
        v
+----------------+
|   HeapTable    |  <-- УПАКОВЩИК/РАСПАКОВЩИК: Tuple <--> []byte
+----------------+
        ^
        | (вызывает FetchPage, получает SlottedPage)
        v
+----------------+
| BufferPoolMgr  |
+----------------+
        ^
        | (вызывает InsertTuple/GetTuple)
        v
+----------------+
|   SlottedPage  |  <-- Работает только с []byte
+----------------+
```
