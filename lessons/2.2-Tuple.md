## Часть 1. Итоговая архитектура

### 1. `Value` — интерфейс любого значения

Центральный интерфейс. Любой объект, представляющий значение ячейки таблицы, обязан его реализовывать. Объединяет данные и поведение.

Ключевая идея контракта: **`Value` самоописываем при (де)сериализации**. Он умеет не только записать себя в буфер, но и прочитать себя обратно, сообщив, *сколько байт он занял*. Благодаря этому вышестоящий код (`Tuple`) не обязан знать, фиксированной длины тип или переменной — он просто двигается по байтам на величину, которую вернул сам `Value`.

```go
// TypeID — лёгкое перечисление типов (1 байт: хватает с запасом, дёшево хранить в схеме).
type TypeID uint8

const (
    INVALID TypeID = iota
    INTEGER
    TEXT
)

// Value — интерфейс для любого типизированного значения.
// NULL за пределами Value: отсутствие значения кодируется как nil-элемент
// в Tuple.values, а сюда NULL никогда не попадает.
type Value interface {
    // String возвращает человекочитаемое представление значения (для REPL/логов).
    // Идиоматичный fmt.Stringer — fmt.Println работает автоматически.
    String() string

    // Serialize дописывает байтовое представление значения в конец buf
    // и возвращает расширенный срез. Для типов переменной длины значение
    // само пишет свой length-prefix — буфер остаётся самоописываемым.
    // Append-style избавляет от аллокации на каждое значение при сборке кортежа.
    Serialize(buf []byte) []byte

    // Deserialize заполняет получатель из начала buf и возвращает число
    // прочитанных байт, чтобы вызывающий мог сдвинуться к следующему значению.
    // Вызывается на уже созданном (пустом) значении нужного типа.
    Deserialize(buf []byte) int

    // Compare сравнивает текущий объект с другим.
    // Возвращает -1 (<), 0 (==), 1 (>). NULL'ы должны быть отсеяны выше.
    Compare(other Value) (int, error)
}
```

### 2. Конкретные реализации: `IntegerValue`, `TextValue`, …

Для каждого типа данных — отдельная структура, реализующая `Value`. Типы фиксированной и переменной длины различаются *только* своими `Serialize`/`Deserialize`; для остального кода они неотличимы.

```go
type IntegerValue struct{ value int64 }  // фикс. длина: ровно 8 байт, без префикса
type TextValue    struct{ value string } // перем. длина: несёт свой uint32 length-prefix
```

- `IntegerValue` — пишет/читает свои 8 байт; `Deserialize` возвращает `8`.
- `TextValue` — сам предваряет данные `uint32`-длиной (4 байта); `Deserialize` возвращает `4 + len`.

### 3. Фабрика типов: `createValue`

Решает проблему десериализации: чтобы прочитать байты обратно, нужно сперва понять, *какого типа* объект создавать. Тип берётся из схемы (`Column.TypeID`), а сама фабрика — простой `switch`, создающий **пустое** значение нужного типа; заполняет его уже `Deserialize`.

```go
// createValue по TypeID возвращает пустое Value нужного типа, готовое к Deserialize.
func createValue(t TypeID) (Value, error)
```

### 4. `Schema` — чертёж таблицы

Хранит только лёгкие `TypeID`, а не тяжёлые объекты типов.

```go
type Column struct {
    Name   string
    TypeID TypeID // ключ для createValue при распаковке
}

type Schema struct {
    columns []Column
}
```

**Где хранится схема.** Объекты `Schema` живут в памяти. Их персистентное представление — в **Системном Каталоге** (отдельные служебные таблицы внутри самой БД, тема Модуля 2.4).

### 5. `Tuple` — логическое представление строки

Кортеж — это срез значений `[]Value` плюс ссылка на схему. Отдельного массива `isNull` нет: **NULL кодируется нулевым элементом среза** — `values[i] == nil`. Это устраняет дублирование (два параллельных массива, которые надо держать согласованными) ценой того, что `nil`-проверка размазана по коду доступа.

```go
type Tuple struct {
    schema *Schema // ссылка на схему для контекста
    values []Value // values[i] == nil означает NULL в i-й колонке
}

func NewTuple(sc *Schema, vs []Value) *Tuple {
    return &Tuple{schema: sc, values: vs}
}

// Get возвращает значение колонки (или nil, если там NULL).
func (t *Tuple) Get(i int) Value {
    return t.values[i]
}

func (t *Tuple) IsNull(i int) bool {
    return t.values[i] == nil
}
```

## Часть 2. Физическая сериализация `Tuple` → `[]byte`

Формат с заголовком, по образцу PostgreSQL, но сильно упрощённый на старте.

### Структура байтов на диске

```
[ RESERVE | NULL BITMAP (M байт) | ДАННЫЕ (всё остальное) ]
```

**Резервный префикс**
- Назначение: место под будущие метаданные кортежа (в перспективе — заголовок MVCC: `xmin`, `xmax`, `infomask` и т.п.).

**Null Bitmap**
- Размер: 1 бит на колонку схемы → `M = (N-1)/8 + 1` байт для `N` колонок (1–8 колонок — 1 байт, 9–16 — 2 байта, и т.д.).
- Если `i`-й бит = 1 → `i`-я колонка `NULL`. Это позволяет вообще не хранить `NULL`-значения в области данных.
- При сериализации бит выставляется там, где `values[i] == nil`; при десериализации выставленный бит превращается обратно в `nil`-элемент.

**Область данных (PG-style, self-describing values)**
- Значения уложены строго **в порядке колонок схемы**, среди не-NULL колонок.
- Каждый `Value` сериализует себя **целиком и самодостаточно**: тип фиксированной длины (`INTEGER`) пишет ровно свои байты; тип переменной длины (`TEXT`) сам предваряет данные `uint32`-длиной (LittleEndian, 4 байта). Цикл сериализации `Tuple` *не разветвляется* на «фикс/varlen» — это знание инкапсулировано в самих значениях.
- NULL-значения не занимают места в области данных вообще.

**Цена выбора.** Доступ к k-й колонке при распаковке — O(k): чтобы добраться до неё, надо последовательно «пройти» предыдущие значения (каждое сообщает свою длину при чтении). На наших масштабах это не проблема; при необходимости лечится кэшем смещений в `Tuple` после первого скана (как делает PG в `TupleTableSlot`).

### Интерфейс сериализации

Преобразование `Tuple` ↔ `[]byte` задаётся двумя точками входа; сами алгоритмы — задача реализации, не плана.

```go
func (t *Tuple) Serialize() []byte                          // Tuple → []byte (формат выше)
func NewTupleFromBytes(sc *Schema, b []byte) (*Tuple, error) // []byte → Tuple (через createValue + Deserialize)
```

## Часть 3. Зоны ответственности и взаимодействие слоёв

> **Контекст для следующего урока.** `HeapTable` и `TableAccessMethod` будут реализованы в Занятии 2.3. Здесь описаны только для того, чтобы было ясно, *кто* и *как* будет использовать `Tuple`/`Value` — это влияет на их контракт.

### Клиент: `Executor` — но через посредника

Конечный потребитель — `Executor`. Он работает с логическими данными:
- При `INSERT` — создаёт `Tuple` из `Value`.
- При `SELECT ... WHERE` — получает `Tuple` и сравнивает его `Value`.

Но `Executor` **не должен** знать, как `Tuple` лежит на диске. `SlottedPage` или что-то другое — ему неважно. Он работает через абстрактный контракт **`TableAccessMethod`**.

### Посредник: `HeapTable` — реализация `TableAccessMethod`

`HeapTable` — мост между логическим миром `Executor` и физическим миром `SlottedPage`. Реализует интерфейс с методами:

```go
InsertTuple(tuple *Tuple) (TID, error)
GetTuple(tid TID) (*Tuple, error)
NewIterator() *TableIterator
```

Ключевой момент: `HeapTable` **принимает и возвращает логические `Tuple`, а не сырые `[]byte`**.

### Упаковщик: `HeapTable`

Именно `HeapTable` выполняет всю «грязную» работу превращения логики в физику и обратно.

**Поток `INSERT` (сверху вниз):**

1. **Executor:** создаёт логический `*Tuple` со срезом `[]Value` и передаёт в `HeapTable`.
   ```go
   tupleToInsert := NewTuple(schema, []Value{
       &IntegerValue{42},
       &TextValue{"Bob"},
   })
   heapTable.InsertTuple(tupleToInsert)
   ```
2. **HeapTable:**
    - Получает `*Tuple`. Здесь происходит **упаковка**: `tuple.Serialize()` (заголовок, null bitmap, укладка данных) — получается `finalTupleBytes []byte`.
    - Обращается к `BufferPoolManager` за страницей с достаточным свободным местом.
    - Вызывает `slottedPage.InsertTuple(finalTupleBytes)`.
3. **SlottedPage:** получает `[]byte` и сохраняет, ничего не зная об их смысле.

**Поток `SELECT` (снизу вверх):**

1. **Executor:** запрашивает у `HeapTable` итератор для сканирования.
2. **TableIterator** (созданный `HeapTable`):
    - Через `BufferPoolManager` получает страницу, оборачивает в `*SlottedPage`.
    - Вызывает `slottedPage.GetTuple(slotID)` → получает `[]byte`.
    - Здесь происходит **распаковка**: `NewTupleFromBytes(schema, bytes)` через `createValue` + `Deserialize` восстанавливает `*Tuple` со срезом `[]Value` (с `nil` на месте NULL).
3. **Executor:** получает готовый логический `*Tuple` и работает с его значениями.

### Схема взаимодействия

```
+----------------+
|    Executor    |  <-- Работает с логическими Tuple, Value
+----------------+
        ^
        | (вызывает методы интерфейса TableAccessMethod)
        v
+----------------+
|   HeapTable    |  <-- УПАКОВЩИК/РАСПАКОВЩИК: Tuple <--> []byte
+----------------+
        ^
        | (вызывает FetchPage, получает SlottedPage)
        v
+----------------+
| BufferPoolMgr  |
+----------------+
        ^
        | (вызывает InsertTuple/GetTuple)
        v
+----------------+
|   SlottedPage  |  <-- Работает только с []byte
+----------------+
```

## Часть 4. Тестовые сценарии

Философия — начать с малого: один содержательный round-trip и точечная проверка `Compare`. Эталона байтов руками не выписываем — свойство `Deserialize(Serialize(x)) == x` само себе oracle.

### 1. Round-trip кортежа (`Serialize` → `NewTupleFromBytes`)

Собрать кортеж на **9 колонок** (это минимум, при котором null bitmap занимает 2 байта — проверяем арифметику сдвигов `i/8`, `i%8` через границу байта), сериализовать, десериализовать и выборочно сверить значения колонок и `IsNull`.

Условия, делающие сценарий показательным:
- **смешанные типы** `INTEGER` и `TEXT` в колонках — проверяем и фикс-длину, и varlen с length-prefix;
- **1–2 колонки NULL, причём одна из них — колонка с индексом 8** (9-я): только так выставляется и читается *второй* байт bitmap'а, иначе арифметика перехода 8→9 не задействуется;
- **TEXT-колонка стоит перед последующими не-NULL колонками**: неверный сдвиг по данным (или потерянный length-prefix) тогда испортит чтение *следующей* колонки и тест упадёт.

### 2. `Compare` для обоих типов

- `IntegerValue` и `TextValue`: случаи `<`, `==`, `>` → ожидаемые `-1 / 0 / 1`.
- Сравнение **разных типов** (`INTEGER` против `TEXT`) → возвращается ошибка (отдельная ветка в каждом методе).
